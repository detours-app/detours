Summary

The audit ran five independent review dimensions (logic, bugs, security, architecture, conventions) over the whole codebase, then scored every finding with a separate agent that re-read the actual file. Twenty-one findings cleared a confidence threshold of 80; after merging duplicates that different dimensions reported against the same root cause, fourteen distinct issues remain. The architecture findings (the 3,244-line FileOperationQueue god class, the four-way duplicated conflict-resolution loop) scored below threshold and are not listed.

The dominant theme is deletion safety. The project's absolute rule — user files are never permanently removed, everything goes to Trash with undo — is broken in four places: the local "Replace" path in the copy/move conflict dialog, and three server-side operations (copy, move, upload) that call removeItem on a client-supplied destination with no containment check before writing. On the remote helper these are silent (try?), so a client bug or a crafted path destroys data with no trace and no recovery.

The second theme is the new remote/RPC layer. The daemon's request loop has no per-request error handling, so any ordinary filesystem error (a missing file, an unwatch after reconnect) crashes the whole process. The client's request loop hangs forever when the server returns an error-kind envelope. The frame reader allocates whatever a 4-byte length field claims, with no cap on the client side. The add-host "Test Connection" button is a stub that always reports success, and the SSH host-key trust file the app writes is never actually passed to ssh. Nothing here is shipped yet — it lives on feature/remote-vm-browsing — but it is not ready as it stands.
